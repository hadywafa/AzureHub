# Microsoft Entra PIM in Azure

**PIM — Privileged Identity Management** gives users temporary, controlled access to privileged roles instead of granting permanent access.

The best mental model is:

```text
Azure RBAC decides:
- What can you do?
- Where can you do it?

PIM decides:
- When can you use that permission?
- For how long?
- Does it require MFA, justification, a ticket, or approval?
```

PIM is part of **Microsoft Entra ID Governance** and generally requires Microsoft Entra ID P2 or Microsoft Entra ID Governance licensing. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-configure?utm_source=chatgpt.com 'What is Privileged Identity Management? - Microsoft Entra ID Governance | Microsoft Learn'))

---

# Normal RBAC versus PIM

## Permanent RBAC assignment

```text
User: Hady
Role: Contributor
Scope: PROD Subscription
Assignment: Active
Duration: Permanent
```

Hady can modify resources in the production subscription at any time.

## PIM eligible assignment

```text
User: Hady
Role: Contributor
Scope: PROD Subscription
Assignment: Eligible
```

Hady does **not** normally have Contributor access.

When production work is required, Hady activates the role:

```text
Eligible
   │
   │ Activate for 2 hours
   ▼
Active Contributor
   │
   │ 2 hours expire
   ▼
Eligible again
```

PIM can require MFA, justification, ticket information and approval before activating the role. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-resource-roles-assign-roles?utm_source=chatgpt.com 'Assign Azure resource roles in Privileged Identity Management - Microsoft Entra ID Governance | Microsoft Learn'))

# Real production example

Assume your company has:

```text
Subscription: B2B-PROD-SUB
Resource Group: RG-B2B-PROD-01
User: hady@company.com
```

You deploy applications to resources inside that resource group.

The security team gives you:

```text
Role: Contributor
Scope: RG-B2B-PROD-01
Assignment type: Eligible
Eligibility: Permanent
```

The activation policy is:

```text
Maximum activation: 2 hours
MFA: Required
Justification: Required
Ticket number: Required
Approval: Required
Approvers: Platform Team Leads
```

Your standing access is therefore:

```text
Normally:
No Contributor permission

During approved activation:
Contributor on RG-B2B-PROD-01 for two hours

After two hours:
Contributor permission is automatically removed
```

You do not receive access to other resource groups or the entire subscription because the eligible role is scoped only to `RG-B2B-PROD-01`.

# Part 1: How an administrator configures PIM

PIM itself is not created inside every subscription like a resource. It is a Microsoft Entra capability used to manage Azure RBAC assignments.

An administrator needs appropriate resource permissions, normally:

```text
Owner
or
User Access Administrator
```

at the relevant Azure scope. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-resource-roles-assign-roles?utm_source=chatgpt.com 'Assign Azure resource roles in Privileged Identity Management - Microsoft Entra ID Governance | Microsoft Learn'))

## Step 1: Open the Azure resource in PIM

The administrator opens:

```text
Microsoft Entra admin center
→ ID Governance
→ Privileged Identity Management
→ Azure resources
```

Then selects:

```text
B2B-PROD-SUB
```

or a resource group inside it.

The administrator can also create an eligible assignment directly from:

```text
Azure portal
→ Subscription or Resource Group
→ Access control (IAM)
→ Add role assignment
```

PIM is now integrated with Azure RBAC role-assignment pages. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/role-based-access-control/pim-integration?utm_source=chatgpt.com 'Eligible and time-bound role assignments in Azure RBAC | Microsoft Learn'))

---

## Step 2: Configure the activation policy

The administrator selects:

```text
Azure resources
→ B2B-PROD-SUB
→ Settings
→ Contributor
→ Edit
```

They configure settings such as:

```text
Activation maximum duration: 2 hours
Require MFA: Yes
Require justification: Yes
Require ticket information: Yes
Require approval: Yes
Approvers: Platform Team Leads
```

PIM settings are configured **per role and per resource**.

For example:

```text
Contributor on B2B-PROD-SUB
```

can have different activation settings from:

```text
Owner on B2B-PROD-SUB
```

Also, settings configured at subscription scope are not automatically inherited by resource-group-level PIM policies. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-resource-roles-configure-role-settings?utm_source=chatgpt.com 'Configure Azure resource role settings in PIM - Microsoft Entra ID Governance | Microsoft Learn'))

---

## Step 3: Create the eligible assignment

The administrator selects:

```text
Assignments
→ Add assignments
```

Then configures:

```text
Role: Contributor
Member: hady@company.com
Assignment type: Eligible
Scope: RG-B2B-PROD-01
```

The assignment duration can be:

```text
Permanent eligible
```

or:

```text
Eligible from July 1 until December 31
```

Then the administrator selects:

```text
Assign
```

Eligible assignments require the user to activate the role before using its permissions. An active assignment, by contrast, is immediately usable and does not require activation. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-resource-roles-assign-roles?utm_source=chatgpt.com 'Assign Azure resource roles in Privileged Identity Management - Microsoft Entra ID Governance | Microsoft Learn'))

# Part 2: How you activate your role

When you need production access, open:

```text
Microsoft Entra admin center
→ ID Governance
→ Privileged Identity Management
→ My roles
→ Azure resources
→ Eligible assignments
```

You should see something like:

| Role        | Resource       | Scope          | Status   |
| ----------- | -------------- | -------------- | -------- |
| Contributor | RG-B2B-PROD-01 | Resource group | Eligible |

Select:

```text
Activate
```

You may also be able to activate eligible access from the Azure portal's **Access control (IAM)** page, depending on the interface available in your tenant. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/role-based-access-control/role-assignments-eligible-activate?utm_source=chatgpt.com 'Activate eligible Azure role assignments - Azure RBAC | Microsoft Learn'))

---

## Complete the activation request

Enter:

```text
Duration: 2 hours

Justification:
Deploy release 2026.06.30 to B2B production

Ticket:
CHG0012456
```

You may then be required to complete MFA.

If approval is configured, the request becomes:

```text
Pending approval
```

An approver receives the request and approves or denies it. After approval, your role becomes active for the requested duration, within the maximum duration allowed by the PIM policy. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-configure?utm_source=chatgpt.com 'What is Privileged Identity Management? - Microsoft Entra ID Governance | Microsoft Learn'))

# Part 3: How you use the access

After activation, you use Azure normally.

## Azure portal

You can open:

```text
RG-B2B-PROD-01
```

and create, update or delete resources permitted by the Contributor role.

## Azure CLI

After activating PIM:

```bash
az login

az account set \
  --subscription "B2B-PROD-SUB"

az group show \
  --name "RG-B2B-PROD-01"
```

For example, you could restart a web app:

```bash
az webapp restart \
  --resource-group "RG-B2B-PROD-01" \
  --name "b2b-prod-api"
```

PIM does not provide a separate version of Azure CLI. It temporarily makes the Azure RBAC assignment active, and then the portal, CLI, PowerShell, Terraform or other tools use that active permission.

# Activating at a narrower scope

Suppose your eligible assignment is:

```text
Contributor
Scope: Entire PROD subscription
```

During activation, PIM can allow you to narrow the activation to a particular resource group:

```text
Eligible scope:
B2B-PROD-SUB

Activated scope:
RG-B2B-PROD-01
```

That means you activate only the access required for the task rather than using Contributor across the entire subscription. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/role-based-access-control/role-assignments-eligible-activate?utm_source=chatgpt.com 'Activate eligible Azure role assignments - Azure RBAC | Microsoft Learn'))

# What happens when the duration expires?

Assume activation started at:

```text
10:00 AM
```

with:

```text
Duration: 2 hours
```

At approximately:

```text
12:00 PM
```

the active role expires:

```text
Contributor — Active
        ↓
Contributor — Eligible
```

You no longer have Contributor permission. To continue working, you must submit another activation request.

You can also manually deactivate the role when the task finishes instead of waiting for it to expire.

# Eligible, active and permanent active

| Assignment type        | Meaning                                                        |
| ---------------------- | -------------------------------------------------------------- |
| **Eligible**           | User can activate the role when needed                         |
| **Active time-bound**  | Role is active only between specified dates or times           |
| **Permanent active**   | User always has the role without activation                    |
| **Permanent eligible** | User remains eligible indefinitely but must activate each time |

The secure production preference is usually:

```text
Permanent eligible
+
Short activation duration
+
MFA
+
Justification
+
Approval for highly privileged roles
```

# PIM manages three different areas

```text
Microsoft Entra PIM
│
├── Microsoft Entra roles
│   ├── Global Administrator
│   ├── User Administrator
│   └── Privileged Role Administrator
│
├── Azure resource roles
│   ├── Owner
│   ├── Contributor
│   ├── Reader
│   └── AKS RBAC Admin
│
└── PIM for Groups
    ├── Eligible group member
    └── Eligible group owner
```

For your question about access to a subscription or resource group, you are using:

```text
PIM
→ Azure resources
→ Azure RBAC role
```

That is different from activating an Entra directory role such as Global Administrator.

---

# Important limitation for pipelines

A human user can be made eligible because the user can:

```text
Perform MFA
Enter justification
Request approval
Activate the role
```

A service principal or managed identity cannot normally be assigned an eligible role because it cannot perform this interactive activation process. For automation identities, use carefully scoped active assignments, potentially time-bound, rather than user-style eligible activation. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/role-based-access-control/pim-integration?utm_source=chatgpt.com 'Eligible and time-bound role assignments in Azure RBAC | Microsoft Learn'))

# Final mental model

```text
Hady
  │
  ├── Eligible Contributor
  │       Scope: RG-B2B-PROD-01
  │
  ├── Activates role
  │       MFA
  │       Justification
  │       Change ticket
  │       Manager approval
  │
  ├── Active Contributor for 2 hours
  │
  └── Role automatically expires
```

So PIM is not the permission itself. The permission is an **Azure RBAC role**. PIM controls when and under what conditions you are allowed to activate that permission.
