# Microsoft Entra Entitlement Management

**Entitlement Management controls how users request, receive, review, and lose access to a bundle of resources.**

The simplest mental model is:

```text
Entitlement Management:
- Who should receive access?
- What package of access should they receive?
- Who must approve it?
- How long should they keep it?

PIM:
- When can the user activate privileged access?
- For how many hours?
- What security checks are required?
```

---

# PIM versus Entitlement Management

| Area             | Entitlement Management                         | PIM                                             |
| ---------------- | ---------------------------------------------- | ----------------------------------------------- |
| Main purpose     | Access request and lifecycle                   | Just-in-time privileged access                  |
| Main object      | Access package                                 | Eligible role assignment                        |
| Typical duration | Days, months, project duration                 | Minutes or hours per activation                 |
| Access included  | Groups, Teams, applications, SharePoint, roles | Entra roles, Azure RBAC roles, group membership |
| User action      | Request the package                            | Activate an eligible role                       |
| Approval         | Approval to receive the package                | Approval to activate privilege                  |
| Expiration       | Removes package resources                      | Deactivates the role                            |
| Best use         | Onboarding users to a project or department    | Performing privileged administration            |

An **access package** is a bundle of resources that someone needs for a project or job. It can contain security-group membership, Microsoft 365 groups or Teams, enterprise-application roles, SharePoint site membership and certain role assignments. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-overview 'What is entitlement management? - Microsoft Entra ID Governance | Microsoft Learn'))

# Main components

```text
Entitlement Management
│
├── Catalog
│     Container holding resources
│
├── Access Package
│     Bundle of access given to a user
│
├── Resources and Resource Roles
│     Group Member
│     Application User
│     SharePoint Member
│     etc.
│
├── Policy
│     Who can request?
│     Is approval required?
│     Who approves?
│     How long does access last?
│
└── Assignment
      The actual package assigned to a user
```

## 1. Catalog

A **catalog** is an administrative container for related resources and access packages.

Example:

```text
Catalog: B2B Platform
```

It might contain:

```text
B2B-UAT-Engineers group
B2B-PROD-Support group
Grafana enterprise application
B2B SharePoint site
B2B Teams team
```

## 2. Access package

An access package combines the specific resource roles the user needs.

Example:

```text
Access package: B2B Production Support Engineer
```

Containing:

```text
B2B-PROD-SUPPORT group       → Member
Grafana application          → Viewer
B2B Production SharePoint    → Member
B2B Support Team             → Member
```

## 3. Assignment policy

The policy controls how the package is obtained:

```text
Who can request: Members of DevOps department
Approval: Manager + Platform owner
Duration: 90 days
Extension: Allowed
Access review: Every 30 days
```

A package can have multiple policies—for example, one policy for employees and another with stricter approvals for external contractors. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-request-policy?utm_source=chatgpt.com 'Change request settings for an access package in ...'))

# Complete practical example

Assume your company has a B2B project.

A new DevOps engineer needs:

```text
Access to the B2B Teams team
Read access to Grafana
Access to project documentation
Azure Reader access to the UAT resource group
Eligibility for Contributor access in production
```

Without Entitlement Management, administrators would manually:

```text
Add the user to multiple groups
Assign application access
Add SharePoint permissions
Assign Azure permissions
Remember to remove everything when the project ends
```

This is difficult to audit and easy to forget.

Instead, create:

```text
Access package:
B2B DevOps Engineer
```

Containing:

```text
B2B-DEVOPS group                     → Member
B2B Grafana application              → Viewer
B2B SharePoint site                  → Member
B2B-UAT-READERS group                → Member
B2B-PROD-CONTRIBUTORS-PIM group      → Eligible Member
```

The policy could be:

```text
Requestors: Internal employees in Engineering
Approval stage 1: Requestor's manager
Approval stage 2: B2B platform owner
Assignment duration: 90 days
Access review: Every 30 days
Extension: Requires approval
```

The workflow becomes:

```text
Hady requests "B2B DevOps Engineer"
                │
                ▼
Manager approves
                │
                ▼
Platform owner approves
                │
                ▼
Hady automatically receives:
- Group memberships
- Application access
- SharePoint access
- Azure UAT Reader access
- Eligible production access
                │
                ▼
90 days expire
                │
                ▼
Access is automatically removed
```

When a group is included as a resource, Entitlement Management adds the approved user to that group. When the package assignment expires, it removes the user unless another access-package assignment still requires the same membership. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-resources?utm_source=chatgpt.com 'Change resource roles for an access package in entitlement management - Microsoft Entra ID Governance | Microsoft Learn'))

# How an administrator creates it

Open:

```text
Microsoft Entra admin center
→ ID Governance
→ Entitlement management
```

## Step 1: Create a catalog

Go to:

```text
Entitlement management
→ Catalogs
→ New catalog
```

Configure:

```text
Name: B2B Platform
Description: Resources required by B2B project teams
Enabled: Yes
Enabled for external users: Only if required
```

## Step 2: Add resources to the catalog

Add resources such as:

```text
Groups and Teams
Enterprise applications
SharePoint Online sites
```

For example:

```text
Group: B2B-DEVOPS
Application: Grafana
SharePoint: B2B Project Site
```

## Step 3: Create an access package

Go to:

```text
ID Governance
→ Entitlement management
→ Access packages
→ New access package
```

Configure:

```text
Name: B2B DevOps Engineer
Catalog: B2B Platform
```

## Step 4: Add resource roles

Select:

```text
Resource                  Role
------------------------------------------------
B2B-DEVOPS group          Member
Grafana application       Viewer
B2B Project site          Member
B2B Teams team            Member
```

Access packages are created inside catalogs, and each resource must be assigned an appropriate resource role such as group member, group owner or application role. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-first 'Tutorial - Manage access to resources in entitlement management - Microsoft Entra ID Governance | Microsoft Learn'))

## Step 5: Configure who can request

For example:

```text
Users who can request:
Specific users and groups

Allowed group:
Engineering-Employees
```

Other possibilities include:

```text
All members in the directory
Specific connected organizations
External users
Administrator assignment only
```

## Step 6: Configure approval

Example:

```text
Approval required: Yes

First approver:
Requestor's manager

Second approver:
B2B Platform Owners

Approver decision timeout:
14 days

Justification required:
Yes
```

Approvers can include specific users, members of a group, the requestor’s manager, or internal/external sponsors, depending on the request policy. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-approval-policy?utm_source=chatgpt.com 'Change approval settings for an access package in entitlement management - Microsoft Entra - Microsoft Entra ID Governance | Microsoft Learn'))

## Step 7: Configure lifecycle

Example:

```text
Assignment expires after: 90 days
Allow extension: Yes
Require approval for extension: Yes
```

Expiration can be based on a specific date, a number of days or hours, or configured not to expire, although time-limited access is preferable for project and external access. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-lifecycle-policy?utm_source=chatgpt.com 'Change lifecycle settings for an access package in ...'))

## Step 8: Configure access reviews

For example:

```text
Review frequency: Every 30 days
Reviewer: User's manager
If reviewer does not respond: Remove access
```

This answers:

```text
Does Hady still work on B2B?
Does he still need this package?
Should his access continue?
```

# How you request and use the package

As the user, open the Microsoft **My Access** portal.

Then:

```text
Access packages
→ Find "B2B DevOps Engineer"
→ Request
```

Enter information such as:

```text
Business justification:
I am joining the B2B DevOps team and need access
to support UAT and production releases.

Requested duration:
90 days
```

After approval:

```text
Package status: Delivered
```

You are automatically added to the configured groups, applications and sites. You do not need to request each resource separately. Users request packages through My Access, and approvers can approve or deny those requests from the same experience. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-request-access?utm_source=chatgpt.com 'Request an access package - entitlement management - Microsoft Entra ID Governance | Microsoft Learn'))

# How it provides Azure subscription or resource-group access

A common, mature implementation is:

```text
Azure RBAC assignment:
B2B-UAT-READERS group
    │
    └── Reader on RG-B2B-UAT-01
```

Then place that group in the access package:

```text
Access package:
B2B UAT Engineer
    │
    └── B2B-UAT-READERS group → Member
```

The result is:

```text
User receives access package
        ↓
User becomes a group member
        ↓
Group already has Azure Reader
        ↓
User receives Reader on RG-B2B-UAT-01
```

Microsoft explicitly supports using a security group in an access package and assigning an Azure RBAC role to that group. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-overview 'What is entitlement management? - Microsoft Entra ID Governance | Microsoft Learn'))

## Direct Azure RBAC support

As of June 2026, Microsoft also documents **direct Azure RBAC roles in access packages as Preview**. It supports management-group, subscription and resource-group scopes, with active or eligible Azure role assignments. Because this remains a preview feature, group-based Azure RBAC is still the more established design for production environments. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-azure-role-assignments 'Assign Azure Role-based access control (RBAC) Roles - Entitlement management (Preview) - Microsoft Entra ID Governance | Microsoft Learn'))

# Combining Entitlement Management with PIM

This is the strongest design:

```text
Entitlement Management:
Determines whether Hady belongs to the B2B project
and should have access for the next 90 days.

PIM:
Determines when Hady may activate production
Contributor for two hours.
```

Example:

```text
Access package assignment:
B2B DevOps Engineer — 90 days
        │
        └── Eligible member:
            B2B-PROD-CONTRIBUTORS-PIM
                    │
                    └── Contributor:
                        RG-B2B-PROD-01
```

Your normal access:

```text
Reader or no write access to production
```

When deploying:

```text
PIM
→ My roles
→ Groups or Azure resources
→ Activate
→ 2 hours
→ MFA
→ Change ticket
→ Approval
```

After two hours:

```text
Contributor access is deactivated
```

After 90 days:

```text
The access package expires
→ Your eligibility is removed completely
```

Entitlement Management can provide eligible membership in groups governed by PIM. The user then activates that membership through PIM when privileged access is needed. ([Microsoft Learn](https://learn.microsoft.com/en-us/entra/id-governance/entitlement-management-access-package-eligible?utm_source=chatgpt.com 'Assign eligible group membership and ownership in access packages via Privileged Identity Management for Groups - Microsoft Entra ID Governance | Microsoft Learn'))

# The key distinction

```text
Entitlement Management:
"Should Hady be part of the B2B project?"

PIM:
"Should Hady be allowed to use Contributor right now?"
```

Or more precisely:

```text
Entitlement Management = Access lifecycle

PIM = Privileged-access activation
```

A production-grade design often uses both:

```text
Access package
    │
    ├── Permanent project-level access
    │     Teams, documentation, applications, Reader
    │
    └── PIM eligibility
          Contributor or administrator privileges
          activated only when needed
```
