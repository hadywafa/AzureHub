# Azure Policy: Built-in and Custom Policy Examples

## Azure Policy overview

Azure Policy evaluates Azure resources against organizational rules. A **policy definition** contains the rule, while a **policy assignment** applies that definition to a scope such as a management group, subscription, resource group, or individual resource. Assignments apply to the selected scope and its child resources. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/overview?utm_source=chatgpt.com 'Overview of Azure Policy'))

| Object            | Purpose                                                                  |
| ----------------- | ------------------------------------------------------------------------ |
| Policy definition | Contains the compliance condition and effect                             |
| Policy assignment | Applies a definition to a particular scope                               |
| Initiative        | Groups multiple policy definitions into one governance package           |
| Scope             | Management group, subscription, resource group, or resource              |
| Exclusion         | Child scope that should not receive the assignment                       |
| Parameter         | Makes a definition reusable for different environments                   |
| Remediation task  | Corrects existing resources for `Modify` or `DeployIfNotExists` policies |

### Common policy effects

Each definition has an effect that determines what Azure Policy does when the condition matches. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/concepts/effect-basics 'Azure Policy definitions effect basics - Azure Policy | Microsoft Learn'))

| Effect              | Behaviour                                                      |
| ------------------- | -------------------------------------------------------------- |
| `Audit`             | Allows the operation but reports the resource as non-compliant |
| `Deny`              | Rejects a non-compliant create or update request               |
| `Modify`            | Adds, changes, or removes supported properties or tags         |
| `DeployIfNotExists` | Deploys a related resource or configuration when it is missing |
| `AuditIfNotExists`  | Audits whether a related resource or configuration exists      |
| `Disabled`          | Disables evaluation for that assignment                        |

`Modify` and `DeployIfNotExists` assignments require a managed identity with sufficient Azure RBAC permissions when remediation is performed. Existing resources are not automatically changed by a normal evaluation; a remediation task must be created. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/concepts/effect-modify 'Azure Policy definitions modify effect - Azure Policy | Microsoft Learn'))

# 2. Built-in policy example: Allowed locations

## Requirement

The organization permits resource deployment only in:

- UAE North
- UAE Central

Deployments to regions such as East US or West Europe must be denied.

Azure provides a built-in policy named **Allowed locations** for this requirement. Built-in policies should normally be checked before developing a custom definition. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/overview?utm_source=chatgpt.com 'Overview of Azure Policy'))

## Expected result

| Deployment                    | Result                                     |
| ----------------------------- | ------------------------------------------ |
| Resource in `uaenorth`        | Allowed                                    |
| Resource in `uaecentral`      | Allowed                                    |
| Resource in `eastus`          | Denied                                     |
| Existing resource in `eastus` | Reported as non-compliant, but not deleted |

## Portal implementation

### Step 1: Open Azure Policy

1. Sign in to the Azure portal.
2. Search for **Policy**.
3. Select **Assignments**.
4. Select **Assign policy**.

These are the standard Azure portal steps for assigning a built-in policy. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/assign-policy-portal 'Quickstart: Create policy assignment using Azure portal - Azure Policy | Microsoft Learn'))

### Step 2: Configure the scope

For a safe test, assign the policy to a test resource group rather than the entire subscription.

Example:

```text
Subscription: DEV-SUB
Resource group: rg-policy-lab
```

The assignment affects the selected scope and everything below it. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/concepts/definition-structure-basics 'Details of Azure Policy definition structure basics - Azure Policy | Microsoft Learn'))

### Step 3: Select the built-in policy

Under **Policy definition**, search for:

```text
Allowed locations
```

Select the built-in definition.

### Step 4: Configure parameters

Under **Allowed locations**, select:

```text
UAE North
UAE Central
```

### Step 5: Configure enforcement

Set:

```text
Policy enforcement: Enabled
```

No managed identity is required because this policy uses the `Deny` effect rather than `Modify` or `DeployIfNotExists`. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/assign-policy-portal 'Quickstart: Create policy assignment using Azure portal - Azure Policy | Microsoft Learn'))

### Step 6: Add a non-compliance message

Example:

```text
Resources must be deployed in UAE North or UAE Central.
Contact the Cloud Platform team if another region is required.
```

### Step 7: Create the assignment

Review the settings and select **Create**.

## Azure CLI implementation

### Step 1: Define the test scope

```bash
SUBSCRIPTION_ID=$(az account show --query id -o tsv)

RG_NAME="rg-policy-lab"
SCOPE="/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RG_NAME}"
```

Create the test resource group if it does not already exist:

```bash
az group create \
  --name "$RG_NAME" \
  --location uaenorth
```

### Step 2: Find the built-in policy ID

Avoid hardcoding the built-in policy GUID. Retrieve it using its display name:

```bash
POLICY_ID=$(az policy definition list \
  --query "[?policyType=='BuiltIn' && displayName=='Allowed locations'].id | [0]" \
  --output tsv)

echo "$POLICY_ID"
```

Azure CLI supports listing policy definitions and creating assignments using either the definition name or resource ID. ([Microsoft Learn](https://learn.microsoft.com/en-us/cli/azure/policy/definition?view=azure-cli-latest&utm_source=chatgpt.com 'az policy definition | Microsoft Learn'))

### Step 3: Create the assignment

```bash
az policy assignment create \
  --name "allowed-locations-uae" \
  --display-name "Allowed locations - UAE" \
  --description "Allow resources only in UAE North and UAE Central." \
  --scope "$SCOPE" \
  --policy "$POLICY_ID" \
  --params '{
    "listOfAllowedLocations": {
      "value": [
        "uaenorth",
        "uaecentral"
      ]
    }
  }'
```

Azure CLI supports assignment scope, parameters, display names, exclusions and enforcement mode. ([Microsoft Learn](https://learn.microsoft.com/en-us/cli/azure/policy/assignment?view=azure-cli-latest 'az policy assignment | Microsoft Learn'))

## Test the built-in policy

Try creating a storage account in East US:

```bash
TEST_STORAGE="stpolicy${RANDOM}${RANDOM}"

az storage account create \
  --name "$TEST_STORAGE" \
  --resource-group "$RG_NAME" \
  --location eastus \
  --sku Standard_LRS
```

The deployment should fail with a policy violation because `eastus` is not in the assignment’s allowed-location parameter.

A compliant deployment should succeed:

```bash
TEST_STORAGE="stpolicy${RANDOM}${RANDOM}"

az storage account create \
  --name "$TEST_STORAGE" \
  --resource-group "$RG_NAME" \
  --location uaenorth \
  --sku Standard_LRS
```

### Check compliance

In the portal:

```text
Azure Policy
  → Compliance
  → Allowed locations - UAE
```

Or trigger an on-demand scan:

```bash
az policy state trigger-scan \
  --resource-group "$RG_NAME"
```

Azure CLI supports triggering a compliance evaluation at subscription or resource-group scope. ([Microsoft Learn](https://learn.microsoft.com/en-us/cli/azure/policy/state?view=azure-cli-latest 'az policy state | Microsoft Learn'))

View the result:

```bash
az policy state summarize \
  --resource-group "$RG_NAME" \
  --output table
```

# 3. Custom policy example: Require an Environment tag

## Requirement

Every taggable Azure resource must contain an `Environment` tag with one of these values:

```text
Dev
UAT
Prod
```

The policy should initially audit resources. After validation, its effect can be changed to `Deny`.

## Policy design

| Item              | Value                     |
| ----------------- | ------------------------- |
| Target resources  | Resources supporting tags |
| Policy mode       | `Indexed`                 |
| Required tag      | `Environment`             |
| Allowed values    | `Dev`, `UAT`, `Prod`      |
| Initial effect    | `Audit`                   |
| Production effect | `Deny`                    |

For policies targeting resource tags, `Indexed` mode is normally appropriate. Microsoft also recommends identifying the business requirement, mapping it to a resource property or alias, selecting the effect, and then composing the definition. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/tutorials/create-custom-policy-definition 'Tutorial: Create a custom policy definition - Azure Policy | Microsoft Learn'))

---

## Custom policy JSON

Use the following policy definition:

```json
{
  "mode": "Indexed",
  "parameters": {
    "tagName": {
      "type": "String",
      "metadata": {
        "displayName": "Required tag name",
        "description": "Name of the required resource tag."
      },
      "defaultValue": "Environment"
    },
    "allowedTagValues": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed tag values",
        "description": "Values that are permitted for the required tag."
      },
      "defaultValue": [
        "Dev",
        "UAT",
        "Prod"
      ]
    },
    "effect": {
      "type": "String",
      "metadata": {
        "displayName": "Policy effect",
        "description": "Controls whether the policy audits or denies non-compliant resources."
      },
      "allowedValues": [
        "Audit",
        "Deny",
        "Disabled"
      ],
      "defaultValue": "Audit"
    }
  },
  "policyRule": {
    "if": {
      "anyOf": [
        {
          "field": "[concat('tags[', parameters('tagName'), ']')]",
          "exists": "false"
        },
        {
          "field": "[concat('tags[', parameters('tagName'), ']')]",
          "notIn": "[parameters('allowedTagValues')]"
        }
      ]
    },
    "then": {
      "effect": "[parameters('effect')]"
    }
  }
}
```

### How the rule works

```text
IF
    the Environment tag is missing
OR
    the Environment tag value is not Dev, UAT or Prod
THEN
    apply the selected effect
```

Azure Policy definitions use JSON conditions and resource aliases or fields to evaluate resource properties. Parameters allow one definition to be reused with different values or effects. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/concepts/definition-structure-basics 'Details of Azure Policy definition structure basics - Azure Policy | Microsoft Learn'))

# 4. Create the custom policy using the portal

## Step 1: Create the definition

Navigate to:

```text
Azure Policy
  → Definitions
  → + Policy definition
```

Configure:

| Setting             | Example                                                |
| ------------------- | ------------------------------------------------------ |
| Definition location | DEV-SUB or the appropriate management group            |
| Name                | Require Environment tag with allowed values            |
| Description         | Requires resources to have an approved Environment tag |
| Category            | Tags                                                   |
| Policy rule         | Paste the supplied JSON                                |

A definition created at subscription level is available to that subscription. A definition created at management-group level can be assigned to child subscriptions under that management group.

## Step 2: Save the definition

Select **Save**.

## Step 3: Assign the definition

Open the new definition and select **Assign**.

Configure:

```text
Scope: rg-policy-lab
Tag name: Environment
Allowed values: Dev, UAT, Prod
Effect: Audit
```

Use this non-compliance message:

```text
Resources must include the Environment tag with one of these values:
Dev, UAT or Prod.
```

## Step 4: Create the assignment

Select **Review + create**, then **Create**.

# 5. Create the custom policy using Azure CLI

## Step 1: Create the policy rule file

Create `policy-rule.json`:

```json
{
  "if": {
    "anyOf": [
      {
        "field": "[concat('tags[', parameters('tagName'), ']')]",
        "exists": "false"
      },
      {
        "field": "[concat('tags[', parameters('tagName'), ']')]",
        "notIn": "[parameters('allowedTagValues')]"
      }
    ]
  },
  "then": {
    "effect": "[parameters('effect')]"
  }
}
```

## Step 2: Create the parameter file

Create `policy-parameters.json`:

```json
{
  "tagName": {
    "type": "String",
    "metadata": {
      "displayName": "Required tag name",
      "description": "Name of the required resource tag."
    },
    "defaultValue": "Environment"
  },
  "allowedTagValues": {
    "type": "Array",
    "metadata": {
      "displayName": "Allowed tag values",
      "description": "Values permitted for the required tag."
    },
    "defaultValue": [
      "Dev",
      "UAT",
      "Prod"
    ]
  },
  "effect": {
    "type": "String",
    "metadata": {
      "displayName": "Policy effect",
      "description": "Audit, deny or disable the policy."
    },
    "allowedValues": [
      "Audit",
      "Deny",
      "Disabled"
    ],
    "defaultValue": "Audit"
  }
}
```

## Step 3: Create the custom definition

```bash
CUSTOM_POLICY_ID=$(az policy definition create \
  --name "require-environment-tag" \
  --display-name "Require Environment tag with allowed values" \
  --description "Requires resources to contain an approved Environment tag." \
  --mode Indexed \
  --rules "$(cat policy-rule.json)" \
  --params "$(cat policy-parameters.json)" \
  --query id \
  --output tsv)

echo "$CUSTOM_POLICY_ID"
```

Azure CLI supports creating custom definitions with a policy rule, parameters, mode, display name and description. ([Microsoft Learn](https://learn.microsoft.com/en-us/cli/azure/policy/definition?view=azure-cli-latest 'az policy definition | Microsoft Learn'))

## Step 4: Create assignment parameters

Create `assignment-parameters.json`:

```json
{
  "tagName": {
    "value": "Environment"
  },
  "allowedTagValues": {
    "value": [
      "Dev",
      "UAT",
      "Prod"
    ]
  },
  "effect": {
    "value": "Audit"
  }
}
```

## Step 5: Assign the custom policy

```bash
az policy assignment create \
  --name "require-environment-tag-audit" \
  --display-name "Require Environment tag - Audit" \
  --description "Audits resources missing an approved Environment tag." \
  --scope "$SCOPE" \
  --policy "$CUSTOM_POLICY_ID" \
  --params "$(cat assignment-parameters.json)"
```

# 6. Test the custom policy

## Test 1: Resource without the tag

Because the effect is currently `Audit`, the deployment should succeed but later appear as non-compliant.

```bash
TEST_STORAGE="stpolicy${RANDOM}${RANDOM}"

az storage account create \
  --name "$TEST_STORAGE" \
  --resource-group "$RG_NAME" \
  --location uaenorth \
  --sku Standard_LRS
```

Trigger evaluation:

```bash
az policy state trigger-scan \
  --resource-group "$RG_NAME"
```

Check the results:

```bash
az policy state list \
  --resource-group "$RG_NAME" \
  --query "[?policyAssignmentName=='require-environment-tag-audit'].{
    Resource:resourceId,
    State:complianceState,
    Policy:policyDefinitionName
  }" \
  --output table
```

## Test 2: Resource with a valid tag

```bash
TEST_STORAGE="stpolicy${RANDOM}${RANDOM}"

az storage account create \
  --name "$TEST_STORAGE" \
  --resource-group "$RG_NAME" \
  --location uaenorth \
  --sku Standard_LRS \
  --tags Environment=Dev
```

This resource should be compliant.

## Test 3: Change the assignment to Deny

After confirming that the policy identifies the correct resources, update the effect:

```bash
az policy assignment update \
  --name "require-environment-tag-audit" \
  --scope "$SCOPE" \
  --params '{
    "tagName": {
      "value": "Environment"
    },
    "allowedTagValues": {
      "value": [
        "Dev",
        "UAT",
        "Prod"
      ]
    },
    "effect": {
      "value": "Deny"
    }
  }'
```

Now a resource without the tag should be rejected:

```bash
TEST_STORAGE="stpolicy${RANDOM}${RANDOM}"

az storage account create \
  --name "$TEST_STORAGE" \
  --resource-group "$RG_NAME" \
  --location uaenorth \
  --sku Standard_LRS
```

A resource with the approved tag should still succeed:

```bash
TEST_STORAGE="stpolicy${RANDOM}${RANDOM}"

az storage account create \
  --name "$TEST_STORAGE" \
  --resource-group "$RG_NAME" \
  --location uaenorth \
  --sku Standard_LRS \
  --tags Environment=Prod
```

# 7. Optional built-in remediation example

A useful second built-in policy is:

```text
Inherit a tag from the resource group if missing
```

This policy uses the `Modify` effect to copy a tag from the resource group to resources that do not contain it. Existing resources require a remediation task. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/samples/pattern-tags?utm_source=chatgpt.com 'Pattern: Using tags in a policy definition - Azure Policy'))

Example configuration:

```text
Resource group tag:
Environment = UAT

Policy:
Inherit a tag from the resource group if missing

Tag name:
Environment
```

When assigning this policy:

1. Enable creation of a managed identity.
2. Grant the identity the minimum required role.
3. Select **Create a remediation task**.
4. Review the remediation results under Azure Policy → Remediation.

The portal can create the assignment identity and grant the roles declared by the built-in definition. Remediation uses that identity to update existing resources. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/assign-policy-portal 'Quickstart: Create policy assignment using Azure portal - Azure Policy | Microsoft Learn'))

# 8. Recommended production rollout

Use this deployment sequence:

```text
Custom definition
    ↓
Assignment to a non-production resource group
    ↓
Audit effect
    ↓
Review compliance and exceptions
    ↓
Assignment with DoNotEnforce where appropriate
    ↓
Deny or Modify in development subscriptions
    ↓
Gradual rollout to production
```

Microsoft recommends progressive policy rollout so incorrect conditions do not unexpectedly affect critical resources. Azure CLI supports `DoNotEnforce` assignment mode, which allows evaluation without enforcing effects such as deny. ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/governance/policy/how-to/policy-safe-deployment-practices 'Safe deployment of Azure Policy assignments - Azure Policy | Microsoft Learn'))

Example:

```bash
az policy assignment create \
  --name "require-environment-tag-evaluation" \
  --display-name "Require Environment tag - Evaluation" \
  --scope "$SCOPE" \
  --policy "$CUSTOM_POLICY_ID" \
  --params "$(cat assignment-parameters.json)" \
  --enforcement-mode DoNotEnforce
```

# 9. Cleanup

Delete the assignments first:

```bash
az policy assignment delete \
  --name "allowed-locations-uae" \
  --scope "$SCOPE"

az policy assignment delete \
  --name "require-environment-tag-audit" \
  --scope "$SCOPE"
```

Then delete the custom definition:

```bash
az policy definition delete \
  --name "require-environment-tag"
```

Built-in policy definitions cannot be deleted; only their assignments are removed. Custom definitions should not be deleted until their assignments have been removed.
