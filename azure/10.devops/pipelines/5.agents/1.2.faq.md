# ⁉️ **Frequently Asked Questions (FAQs)**

## ❓ **FAQ 1: Can an agent pool contain multiple agents from the same host (VM)?**

**Technically yes**, you can install and register multiple Azure DevOps agents on the same physical or virtual machine, even in the same pool. Each agent runs as a separate process and can have a different name.

**But is it recommended?**  
❌ **No, not in most cases.**  
Here’s why:

- Each agent consumes CPU, memory, and disk I/O. Running multiple agents on one host can cause resource contention and slow builds.
- If the host goes down, you lose all those agents at once.
- It defeats the purpose of horizontal scaling and isolation.

**When might it make sense?**

- If the machine is very powerful and you want to run multiple jobs in parallel without provisioning more VMs.
- For short-term or non-critical workloads.

---

## ❓ **FAQ 2: Can the same host be registered in multiple agent pools?**

✅ **Yes, you can register the same machine as an agent in different pools.**  
Each agent installation is independent, so you can point one agent to `pool-A` and another to `pool-B`.

**What do I think about this practice?**
⚠️ **Generally not recommended for production environments.**  
Here’s why:

- It adds complexity: you now have multiple agents on the same host, tied to different pools.
- Harder to manage updates and troubleshoot failures.
- Risk of overlapping jobs if both pools schedule work on the same host simultaneously.

**Better approach:**

- Use **one agent per host** and assign it to the pool that matches its purpose (e.g., `pool-nonprod-linux` or `pool-prod-linux`).
- If you need flexibility, use **agent tags** and **demands** instead of duplicating agents across pools.

---

### **Why do you see duplicates in your company?**

- Likely because someone installed multiple agents on the same VM with different names (maybe to simulate parallelism or to serve multiple projects).
- When you check hostname/IP, they match because they’re on the same machine.

**Risks of this setup:**

- Resource contention → slower builds.
- Single point of failure → if that VM crashes, multiple agents go offline.
- Harder to audit and maintain.

---

✅ **Best Practice Summary:**

- **One agent per host** (unless you have a strong reason for multiple).
- **One pool per trust boundary/environment** (prod vs non-prod).
- Use **tags and demands** for flexibility instead of duplicating agents.
- For parallelism, scale out with more VMs or use **VM Scale Set agents** (ephemeral, auto-scaling).

## ❓**FAQ 3: What is `http_proxy` in Agent Capabilities?**

- It indicates that the **agent process detected an environment variable named `HTTP_PROXY`** (or `http_proxy`) on the machine.
- This variable is commonly used in enterprise networks to route outbound HTTP traffic through a **proxy server** for security, filtering, or compliance.

### 🔍 Why does Azure DevOps show it as a capability?

- Capabilities are just **key-value pairs** that describe the agent’s environment.
- If `HTTP_PROXY` is set, the agent reports it so that pipelines can **demand** it if needed (though this is rare).
- Example:
  http_proxy = <http://proxy.company.com:8080>

### ⚠️ Impact on Pipelines

- If your agent uses a proxy, all outbound HTTP requests (e.g., `curl`, `npm install`, `helm repo add`) will go through that proxy.
- If the proxy is misconfigured or blocked, tasks may fail (common in secure VNets).
- For HTTPS traffic, you might also see `HTTPS_PROXY` or `NO_PROXY` variables.

### ✅ Best Practices

1. **Set proxy variables only if required** by your network policy.
2. Use `NO_PROXY` for internal endpoints (e.g., `*.azurecr.io`, `*.privatelink.azure.com`) to avoid routing private traffic through the proxy.
3. If you don’t need a proxy, **unset these variables** to avoid unexpected behavior.
4. For Azure DevOps agent service, configure proxy in the agent’s `.env` or system settings if your network enforces it.
