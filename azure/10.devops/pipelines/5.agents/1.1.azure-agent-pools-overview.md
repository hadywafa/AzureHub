# 🖥️ Azure DevOps Agent Pools

## 🔷 What is an Agent Pool?

An **Agent Pool** is a logical grouping of build/release agents that Azure Pipelines uses to run your CI/CD jobs. A **job** is assigned to a pool; an **agent** from that pool picks it up and executes the steps.

- **Microsoft-hosted** pools: ephemeral agents managed by Microsoft (Windows/Linux/macOS). Easiest to start, internet egress required.
- **Self-hosted** pools: agents you manage (VMs, VM Scale Sets, containers, Kubernetes pods). Required in locked-down networks, with custom tools, or for private endpoints.

**Scope**:

- **Organization-level pool** → can be shared across projects (preferred for standardization).
- **Project-level pool** → only visible in that project.

**Security roles on a pool**:

- **Administrator**: manage pool/agents.
- **Service Account**: lets pipelines use the pool (grant this to the project’s build service).
- **Reader**: view-only.

**Where pools are used**:

- **YAML pipelines**: `pool: name: <pool-name>`
- **Classic build/release**: select the pool in the UI per stage.

## 🧭 Core Concepts That Matter

### 1) Agents & Capabilities

- An agent advertises **capabilities** (e.g., `DotNetFramework`, `Java`, custom capabilities).
- Pipelines can require **demands** to ensure the right agent is selected.

```yaml
pool:
  name: "self-hosted-linux"
  demands:
    - kubectl
    - helm
```

### 2) Concurrency & Capacity

- Each agent runs **one job** at a time.
- Organization has a **parallel jobs** limit; add more paid concurrency if needed.
- Scale capacity horizontally (more agents) rather than stacking too much on a single VM.

### 3) Where the Work Runs

- **Workspace**: the directory on agent machine used for source, artifacts, tools.
- Set `clean: resources` or `clean: all` occasionally to avoid bloat and flaky builds.

```yaml
workspace:
  clean: resources
```

### 4) Pool Permissions

- **Least privilege**: grant the **Project Collection Build Service** (org) or **Project Build Service** (project) **Service Account** role on the pool(s) they must use.
- Avoid granting Contributors/Admins broadly on pools.

## 🧩 When to Use Which Pool Type

| Scenario                                                                  | Use This                                                       |
| ------------------------------------------------------------------------- | -------------------------------------------------------------- |
| Public open-source, standard toolchains                                   | Microsoft-hosted                                               |
| Enterprise with **secure VNet**, **private endpoints**, custom toolchains | Self-hosted                                                    |
| Need auto-scale                                                           | Self-hosted **VM Scale Set** pool or **Kubernetes/containers** |
| Strict separation (Prod vs Non-Prod)                                      | Separate pools (and separate agent VMs)                        |

## 🛡️ Best Practices (Enterprise-Ready)

### A. Pool & Agent Design

1. **Separate pools by trust boundary**

   - `pool-nonprod-linux`, `pool-prod-linux`, `pool-nonprod-windows`, etc.
   - Keep production isolated—**separate VMs**, separate service accounts, separate networks.

2. **Specialize pools by OS/toolchain**

   - Avoid “kitchen-sink” images. Clear pools = more predictable builds.

3. **Ephemeral agents for safety & consistency**

   - Use **VM Scale Set** agents or **containerized agents** to provide a fresh environment each run.
   - Advantage: no residue, fewer “works on my agent” problems.
   - Keep a golden image or startup script to bootstrap tools quickly (packer/cloud-init).

4. **Availability & resilience**
   - Run at least **2+ agents per pool**.
   - Distribute across **Availability Zones** if possible.
   - Monitor queue time & job durations; add capacity during peaks.

### B. Security & Governance

1. **Run as least-privileged service account**

   - On Linux: non-root user; on Windows: a least-privileged service account.
   - Disable interactive logon where possible.

2. **Do not store secrets on agents**

   - Use **Azure Key Vault** via service connection; fetch per job.
   - Avoid environment variables in agent startup scripts that contain credentials.

3. **Limit network egress**

   - In secure VNets, allow only the required **Azure DevOps service URLs** and your internal hosts.
   - For private endpoints, ensure DNS resolution is correct from the agent subnet.

4. **Separate prod/non-prod pools** with **separate service connections** and **separate permissions**.

   - Enforce **approvals & checks** at **Environments**/service connections.

5. **Patch & harden**

   - Regularly patch agent VMs/containers.
   - Keep the **agent version up to date** (auto-update is supported).
   - Enable disk encryption, OS hardening baselines, Defender for Cloud.

6. **Workspace hygiene**
   - Use `clean` selectively to reduce contamination; schedule periodic cleanup to manage disk usage.
   - Avoid reusing the same agent for long periods without cleanup.

### C. Performance & Scale

1. **Cache smartly**

   - Use built-in `Cache@2` task for npm/maven/gradle/helm to reduce external fetches.
   - Pin tool versions to avoid frequent installs.

2. **Control parallelism**

   - Set `jobs.<strategy>.maxParallel` thoughtfully.
   - Ensure you have **enough agents** for the parallelism you declare.

3. **Tag agents; use demands**

   - Tag agents with `k8s`, `helmfile`, `java17`, etc.
   - Use `demands` to ensure the right workload lands on the right agent.

4. **Containerized tasks**
   - In YAML, run jobs in a container image (`container:`) to standardize toolchains on a shared host agent.

```yaml
jobs:
  - job: build
    pool: { name: "self-hosted-linux" }
    container: mcr.microsoft.com/devcontainers/base:ubuntu
    steps:
      - script: |
          node -v
          helm version
```

### D. Operations & Monitoring

1. **Monitor queue time & failures**

   - Azure DevOps analytics views; alerts on prolonged queue times.
   - Watch for disk space on agents and network/DNS errors.

2. **Logging & diagnostics**

   - Agent logs (`_diag` folder) for troubleshooting.
   - For pipeline issues in locked networks: validate DNS against custom DNS servers & `/etc/resolv.conf` or systemd-resolved.

3. **Version pinning & auto-update**
   - Enable agent **auto-update** to match server expectations.
   - For VMSS pools, bake the desired agent version into the image and still allow auto-update.

## 🧱 Patterns that Fit Your Environment

Given you’re using **self-hosted agents in a secure VNet** with **custom DNS** and **private endpoints**:

- ✅ Prefer **self-hosted pools** per environment tier: `pool-aks-nonprod-linux`, `pool-aks-prod-linux`.
- ✅ Use **Azure VM Scale Sets** with Managed Identity if you want autoscaling & ephemeral nodes.
  - Allowlist outbound to Azure DevOps service domains and internal endpoints.
  - Ensure DNS forwarders resolve `*.azure.com`, `*.azurecr.io`, storage private endpoints, etc.
- ✅ For **classic release pipelines**, keep the same separation: prod releases should only see **prod pool**.
- ✅ For Helm/AKS work: create a **Kubernetes‑tools** pool with `kubectl`, `helm`, `helmfile`, `az`, `jq`, and set **demands** accordingly.
- ✅ For **private ACR** pulls during CI, either authenticate via managed identity on the agent host or via `az acr login` with a service connection; keep secrets out of the agent.

## 🧪 Common YAML Examples

**Specify a pool:**

```yaml
pool:
  name: "pool-nonprod-linux"
```

**Use demands (based on agent capabilities):**

```yaml
pool:
  name: "pool-k8s-tools"
  demands:
    - kubectl
    - helm
    - java
```

**Clean workspace periodically:**

```yaml
workspace:
  clean: all
```

**Parallel jobs with a matrix:**

```yaml
strategy:
  matrix:
    java17:
      JDK_VERSION: "17"
    java21:
      JDK_VERSION: "21"
```

**Containerized job on a self-hosted agent:**

```yaml
jobs:
  - job: test
    pool: { name: "pool-linux" }
    container: ghcr.io/yourorg/build-images/java17:latest
    steps:
      - script: mvn -v
```

## 🧰 Admin Tips

- **Naming**: `pool-{tier}-{os}-{purpose}` e.g., `pool-prod-linux-k8s`, `pool-nonprod-windows-dotnet`.
- **Tags**: tag agents with `k8s`, `helmfile`, `dotnet7`, `jdk21` and use **demands** in YAML.
- **RBAC**:
  - Grant **Service Account** on pool to the exact project’s build service account.
  - Avoid granting broad Admin on pools to general developers.
- **Checks**: Use **Environments** + **Approvals/Checks** for prod, even with classic releases (or migrate to YAML environments when possible).
- **Agent lifecycle**: treat self-hosted agents as cattle, not pets. Prefer reproducible images and ephemeral runs.

## ❓ Quick FAQ

**Q: Should I share one pool across all projects?**  
A: For orgs with clear separation (prod/non-prod, OS/toolchains), share **org-level pools** but **segment by trust** (separate prod pool). Avoid a single global pool for everything.

**Q: Is VM Scale Set agents worth it?**  
A: Yes, if you need elasticity, ephemeral agents, and reduced toil. They integrate well with Azure DevOps and can auto-scale to demand.

**Q: How do I keep agents up-to-date?**  
A: Enable agent **auto-update** and regularly refresh base images for VMSS/container images.

**Q: Classic vs YAML?**  
A: YAML is recommended for versioned, auditable pipelines. Classic is still widely used—be strict with pool selection and environment approvals for production.
